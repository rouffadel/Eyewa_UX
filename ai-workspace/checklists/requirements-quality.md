# Requirements quality checklist

Use before marking a feature spec `status: approved`.

## Clarity

- [ ] Problem statement is explicit
- [ ] User stories follow As / I want / So that
- [ ] Acceptance criteria are testable (no vague "should work well")
- [ ] Out of scope is listed

## Completeness

- [ ] Dependencies identified
- [ ] Non-functional requirements stated where relevant
- [ ] Edge cases or error behavior addressed

## Traceability

- [ ] Links to raw knowledge or stakeholder input where helpful
- [ ] Conflicts with `decisions/` resolved or noted

## Implementability

- [ ] Plan.md exists or is scheduled
- [ ] Contracts identified for external boundaries
- [ ] Tasks can be derived without guessing
